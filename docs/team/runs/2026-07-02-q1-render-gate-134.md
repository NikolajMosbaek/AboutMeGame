## Sound skip — S4 (#89) and S2 (#87) not pulled

The Sound epics S4 (#89) and S2 (#87) remain below this quality slice, on
**both** charter grounds (docs/team/charter.md, "Standing ordering policy"):

1. **Ordering policy:** "**Sound ranks strictly below all other tracks.**" A
   Sound epic never auto-pulls ahead of feature / quality / graphics work —
   it is promoted only if the user reports audio breakage on a real device.
2. **On-device verification gap (never a silent pass):** iOS-audio behaviours
   — the silent switch and interrupted-audio-state resume — need a real iOS
   device. This desktop-Chromium render gate (headless, SwiftShader) cannot
   prove them, so pulling S4/S2 now could not be verified honestly by the
   gate this run shipped.

## Known limits (stated honestly)

1. **Pre-screenshot failure classes upload no PNG.** The screenshot is taken
   at verify-game.mjs:219, *before* the verdict at line ~243 — so an
   assessor red still ships its visual evidence. But anything that fails
   *earlier* — the in-process vite build failing, the preview readiness gate
   timing out, an `enterWorld` timeout — dies before line 219 and writes no
   PNG; `if-no-files-found: warn` then uploads nothing without stacking a
   second red. The artifact promise is **step-level** (`if: always()` runs
   the upload step) **not file-level** (a PNG is not guaranteed to exist).
   Fixing that needs a last-gasp screenshot in `scripts/`, outside this
   slice's scope.
2. **`WEBGL_ERROR_RE` breadth matches "AudioContext" text.** The frozen
   regex `/webgl|context|THREE/i` matches any console error containing
   "AudioContext" via `/context/i`, so future audio code logging a matching
   console error would red the *render* gate. This is a conscious decision
   (the regex is byte-frozen by the #132 contract), flagged here for the
   future sound epic (S4/S2) to revisit.
3. **Enforcement is procedural-only.** With `main` unprotected (see the
   `gh api` output above — "Branch not protected", empty rulesets), neither
   `build-and-test` nor `render-gate` is a GitHub *required* status check; a
   red gate blocks merge only through the harness's green-only merge
   doctrine, not through platform enforcement.
